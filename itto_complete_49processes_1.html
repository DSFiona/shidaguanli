项目整合管理 — 7个过程
启动1 · 规划1 · 执行2 · 监控2 · 收尾1
整合管理是项目经理最核心的职责，负责把其他九个领域"粘合"在一起统筹协调。打比方：整合管理是乐队指挥，其他领域是各乐手。核心线索：章程→计划→执行→知识→监控→变更控制→收尾。论文必写亮点：变更控制（CCB）和整体计划整合。
启动 1.1 制定项目章程 正式授权项目存在
【干什么】 项目章程是项目的"出生证明"，正式授权项目存在，明确项目目标、项目经理权力及初始资源。没有章程，项目经理无权调配资源。
记忆钩：商业文件→专家+会议→章程+假设日志。
输入
商业文件（商业论证+效益管理计划）协议（合同/MOU）事业环境因素组织过程资产
工具与技术
专家判断数据收集（头脑风暴/焦点小组/访谈）人际关系与团队技能（冲突管理/引导/会议管理）会议
输出
项目章程假设日志
论文套用句式
在项目启动阶段，我依据（甲方合同/商业论证文件），组织启动会议，运用专家判断和访谈收集高层需求，制定了项目章程，明确了项目目标、项目经理权限及初始资源投入，同时建立假设日志记录关键假设，为后续规划奠定基础。
规划 1.2 制定项目管理计划 整合所有子计划的"宪法"
【干什么】 把9个领域的子计划和3大基准（范围/进度/成本）整合成一份总计划，是项目执行的最高纲领。
记忆钩：输入=章程+所有其他规划输出；输出=唯一的"项目管理计划"。
输入
项目章程其他规划过程输出（各子计划）事业环境因素组织过程资产
工具与技术
专家判断数据收集（头脑风暴/核对单/焦点小组/访谈）人际关系与团队技能（冲突管理/引导）会议
输出
项目管理计划（含所有子计划+三大基准）
论文套用句式
在规划阶段，我主持规划会议，运用专家判断将范围、进度、成本、质量、资源、沟通、风险、采购、干系人九个子计划整合，形成了项目管理计划，确立了三大基准（范围基准、进度基准、成本基准），并明确了变更控制流程，作为项目执行的最高纲领。
执行 1.3 指导与管理项目工作 按计划推动实际工作，产出可交付成果
【干什么】 按照计划指挥团队"干活"，产出可交付成果，收集工作绩效数据，记录问题，提出变更请求。
记忆钩：执行→可交付成果+绩效数据+问题日志+变更请求。
输入
项目管理计划项目文件（经验教训/里程碑/沟通/进度/需求跟踪矩阵/风险）批准的变更请求事业环境因素组织过程资产
工具与技术
专家判断项目管理信息系统PMIS会议（项目例会/启动会）
输出
可交付成果工作绩效数据问题日志变更请求项目管理计划更新项目文件更新
论文套用句式
在执行阶段，我依据项目管理计划指导团队开展工作，借助PMIS（Microsoft Project/禅道/Jira）实时收集工作绩效数据。期间共记录问题X项于问题日志，针对（某问题）提出变更请求经CCB审批后执行，最终按期产出（可交付成果名称）。
执行 1.4 管理项目知识 沉淀经验教训，传递隐性知识
【干什么】 利用现有知识并创造新知识，以实现项目目标并促进组织学习。核心输出是经验教训登记册，项目结束时并入组织过程资产。
记忆钩：知识管理+信息管理→经验教训登记册。
输入
项目管理计划项目文件（经验教训/项目团队派工单/资源分解结构/干系人登记册）可交付成果事业环境因素组织过程资产
工具与技术
专家判断知识管理（社交媒体/工作跟踪/会议/专家访谈）信息管理（经验教训库/文档管理系统）人际关系与团队技能（主动倾听/引导/领导力/人际交往/政治意识）
输出
经验教训登记册项目管理计划更新组织过程资产更新
论文套用句式
为促进组织知识积累，我在执行阶段同步开展管理项目知识工作，定期组织团队通过知识分享会议总结阶段经验，将（技术难点解决方案/流程优化措施）等记录于经验教训登记册，项目结束后并入组织过程资产，为后续类似项目提供参考。
监控 1.5 监控项目工作 将绩效数据转化为绩效报告
【干什么】 跟踪、审查和报告项目整体进展，将工作绩效数据加工成工作绩效报告（如仪表盘、状态报告）提交给管理层和干系人。
记忆钩：绩效数据→挣值分析/偏差/趋势→绩效报告+变更请求。
输入
项目管理计划项目文件（假设日志/估算依据/成本预测/问题日志/经验教训/里程碑/质量报告/风险/进度预测）工作绩效信息协议事业环境因素组织过程资产
工具与技术
专家判断数据分析（挣值/偏差/趋势/根因）决策会议
输出
工作绩效报告（仪表盘/状态报告/热点报告）变更请求项目管理计划更新项目文件更新
论文套用句式
在监控阶段，我每周对各领域工作绩效信息进行汇总，运用挣值分析和偏差分析生成工作绩效报告，以项目状态仪表盘形式向（公司高层/甲方）汇报，确保各方及时了解项目状态，并据此提出所需的变更请求。
监控 1.6 实施整体变更控制 CCB审批一切变更，保护基准不被随意修改
【干什么】 通过变更控制委员会（CCB）对所有变更请求进行正式审批，防止未经授权的变更破坏项目基准。批准的变更才能执行，拒绝的变更归档存档。
记忆钩：变更请求→CCB会议→批准/拒绝→更新计划。
输入
项目管理计划（变更管理计划/配置管理计划/范围/进度/成本基准）项目文件（估算依据/需求跟踪矩阵/风险报告）工作绩效报告变更请求事业环境因素组织过程资产
工具与技术
专家判断变更控制工具（变更管理系统）数据分析（备选方案/成本收益）决策（投票/独裁/多标准）会议（变更控制委员会CCB）
输出
批准的变更请求项目管理计划更新项目文件更新
论文套用句式
为防止范围蔓延，我建立了整体变更控制机制，成立由（甲方代表/技术负责人/项目经理）组成的变更控制委员会（CCB）。项目期间共受理变更请求X项，经CCB会议评估对进度、成本、范围的影响后，批准X项、拒绝X项，所有结果更新至项目管理计划，有效保护了三大基准的权威性。
收尾 1.7 结束项目或阶段 正式关闭项目，移交成果，沉淀经验
【干什么】 完成所有收尾活动：获得最终验收、移交产品/服务、释放资源、更新经验教训、存档项目文件。这是整个项目的"终点站"。
记忆钩：验收的可交付成果→收尾→最终报告+移交+组织过程资产更新。
输入
项目章程项目管理计划项目文件（假设日志/估算依据/变更日志/问题日志/经验教训/里程碑/质量控制测量结果/质量报告/需求文件/风险）验收的可交付成果商业文件协议采购文件组织过程资产
工具与技术
专家判断数据分析（文件分析/回归/趋势/偏差分析）会议（收尾会议）
输出
项目文件更新（经验教训登记册更新）最终产品/服务/成果移交最终报告组织过程资产更新
论文套用句式
在项目收尾阶段，我组织召开收尾会议，对照合同和需求逐项确认验收的可交付成果，正式完成最终产品移交。同时编制最终报告，汇总项目绩效数据；将经验教训更新至组织过程资产，释放团队资源，完成项目正式关闭。

项目范围管理 — 6个过程
规划4 · 监控2 · 核心：WBS分解
解决"做什么/不做什么"的问题。核心产出是工作分解结构（WBS）和范围基准。防止范围蔓延（Scope Creep）是重点。注意：核实范围（客户验收）≠ 控制质量（内部检验），先质控再给客户验收。
规划 2.1 规划范围管理 制定如何定义和控制范围的方法
【干什么】 制定范围管理计划和需求管理计划，规定后续范围工作怎么做、WBS怎么建、变更怎么控制等。是范围管理的"元计划"。
记忆钩：输出只有两个计划，工具最简单（专家+数据分析+会议）。
输入
项目章程项目管理计划（质量管理计划/项目生命周期描述/开发方法）事业环境因素组织过程资产
工具与技术
专家判断数据分析（备选方案分析）会议
输出
范围管理计划需求管理计划
论文套用句式
在规划范围管理阶段，我召开规划会议，结合项目特点制定了范围管理计划，明确了WBS的创建方式、范围变更的控制流程；同时制定了需求管理计划，规定需求的收集、优先级排序和变更管理方式，为后续范围工作提供指引。
规划 2.2 收集需求 从干系人处挖掘并记录需求
【干什么】 通过访谈、问卷、原型等手段，把干系人的需求"挖"出来，形成需求文件和需求跟踪矩阵（RTM）。RTM把每条需求和交付物对应，方便后期验收核对。
记忆钩：工具最多（原型法/亲和图/思维导图都在这）；输出=需求文件+RTM。
输入
项目章程项目管理计划（范围/需求/干系人参与计划）项目文件（假设日志/经验教训/干系人登记册）协议事业环境因素组织过程资产
工具与技术
专家判断数据收集（问卷/头脑风暴/访谈/焦点小组）数据分析（文件分析）决策（投票/多标准）数据表现（亲和图/思维导图）人际关系（名义小组）系统交互图原型法标杆对照会议
输出
需求文件需求跟踪矩阵RTM
论文套用句式
在收集需求阶段，我采用访谈和问卷调查向（业务部门/最终用户）收集了X项功能性需求和X项非功能性需求，并利用原型法对关键界面进行演示确认。将成果整理为需求文件，建立需求跟踪矩阵（RTM），确保每项需求可追溯至WBS工作包和最终交付物。
规划 2.3 定义范围 写清楚做什么、不做什么
【干什么】 基于需求文件，详细描述项目和产品的范围，产出项目范围说明书（明确纳入范围和排除范围）。排除范围同样重要，防止"理所当然"的范围蔓延。
记忆钩：需求文件→产品分析→范围说明书。
输入
项目章程项目管理计划（范围管理计划）项目文件（假设日志/需求文件/风险登记册）事业环境因素组织过程资产
工具与技术
专家判断数据分析（备选方案分析）决策（多标准）人际关系与团队技能（引导）产品分析（产品分解/需求分析/系统分析/价值分析）
输出
项目范围说明书（含排除事项）项目文件更新（假设日志/需求文件/需求跟踪矩阵/干系人登记册）
论文套用句式
在定义范围阶段，我运用产品分析技术，对需求文件进行深入分析，形成项目范围说明书。说明书详细描述了项目交付物（包含XX/XX/XX模块），并明确列出了排除范围（如XX功能不在本期范围内），有效防止了后期的范围蔓延。
规划 2.4 创建WBS 将项目逐层分解到可管理的工作包
【干什么】 用分解技术把项目可交付成果层层拆解，形成树状的WBS结构。最底层是"工作包"，可分配责任、估算工时。WBS+WBS词典+范围说明书=范围基准（三合一）。
记忆钩：唯一工具=分解；唯一输出=范围基准。
输入
项目管理计划（范围管理计划）项目文件（项目范围说明书/需求文件）事业环境因素组织过程资产（WBS模板）
工具与技术
专家判断分解（逐层向下至工作包）
输出
范围基准（WBS + WBS词典 + 项目范围说明书）项目文件更新（假设日志/需求文件/需求跟踪矩阵）
论文套用句式
我运用分解技术，将项目可交付成果逐层拆解，创建了工作分解结构（WBS），共分X个层级、X个工作包，涵盖需求分析/设计/开发/测试/部署/培训等阶段。配套编制WBS词典，对每个工作包的验收标准和责任人进行定义，与范围说明书共同构成范围基准，为进度和成本估算提供基础。
监控 2.5 确认范围（核实范围） 让客户正式签字验收可交付成果
【干什么】 将通过内部质量控制的可交付成果提交给客户，通过检查获得客户正式书面验收。注意先后顺序：控制质量（内部）→ 核实范围（客户验收）。
记忆钩：核实的可交付成果→检查→验收的可交付成果（多了"验收"两字）。
输入
项目管理计划（范围/需求管理计划/范围基准）项目文件（经验教训/质量报告/需求文件/需求跟踪矩阵）核实的可交付成果（已过质控）工作绩效数据
工具与技术
检查（审查/产品评审/走查/审计）决策（投票）
输出
验收的可交付成果（客户书面签字）工作绩效信息变更请求项目文件更新
论文套用句式
各阶段可交付成果经内部控制质量合格后，我组织甲方进行确认范围，通过检查和演示，对照需求跟踪矩阵逐项确认功能实现情况。甲方对（XX模块）签署书面验收确认，获得验收的可交付成果，确保交付物满足合同要求。
监控 2.6 控制范围 监控范围偏差，防止范围蔓延
【干什么】 持续监控项目和产品范围的状态，管控范围基准的变更。当实际范围与基准不符时，分析偏差原因，采取纠偏措施或提出变更请求。
记忆钩：工作绩效数据→偏差/趋势分析→工作绩效信息+变更请求。
输入
项目管理计划（范围管理计划/需求管理计划/变更管理计划/配置管理计划/范围基准/绩效测量基准）项目文件（经验教训/需求文件/需求跟踪矩阵）工作绩效数据组织过程资产
工具与技术
数据分析（偏差分析/趋势分析）
输出
工作绩效信息变更请求项目管理计划更新（范围基准/进度基准/成本基准）项目文件更新
论文套用句式
在控制范围过程中，我定期运用偏差分析，对比实际完成工作与范围基准，发现（某干系人私下要求增加XX功能）属于范围蔓延迹象，立即要求通过正式变更控制流程处理，未经批准的工作不纳入执行，有效防止了范围失控。
03
项目进度管理 — 6个过程
规划5 · 监控1 · 核心：关键路径CPM + 进度压缩
解决"什么时候做完"的问题。关键路径（最长路径）决定项目最短工期，关键路径上任何延误=项目延期。进度压缩两招：赶工（加资源，成本增加）和快速跟进（并行，风险增加）。论文必写：CPM+甘特图+进度偏差SV/SPI。
规划 3.1 规划进度管理 制定进度管理的方法和工具
【干什么】 制定进度管理计划，规定用什么方法（CPM/敏捷燃尽图）、什么工具（Project/Excel）来管理进度，以及如何设定基准和报告进度。
记忆钩：输出只有一个=进度管理计划。
输入
项目章程项目管理计划（范围管理计划/开发方法）事业环境因素组织过程资产
工具与技术
专家判断数据分析（备选方案分析）会议
输出
进度管理计划

项目沟通管理 — 3个过程
规划1 · 执行1 · 监控1 · 沟通渠道=n(n-1)/2
解决"信息怎么传、传给谁"的问题。三种沟通方式：交互式（双向实时，如会议/电话）、推式（单向发出，如邮件/报告，不保证理解）、拉式（放共享区被动获取，如知识库）。沟通渠道数量公式：n(n-1)/2，是论文中常用的"数字亮点"。
规划 7.1 规划沟通管理 制定谁需要什么信息、怎么传递的方案
【干什么】 通过沟通需求分析，明确各干系人的信息需求，制定沟通管理计划（沟通矩阵）：什么信息、谁来发、发给谁、用什么方式、多久一次。
记忆钩：规划沟通的核心工具=沟通需求分析（分析信息量=渠道数×内容）。
输入
项目章程项目管理计划（资源/干系人参与计划）项目文件（需求文件/干系人登记册）事业环境因素组织过程资产
工具与技术
专家判断沟通需求分析沟通技术（选择适合的方式）沟通模型（发送→编码→传递→解码→接收→反馈→噪声）沟通方法（交互式/推式/拉式）人际关系与团队技能数据表现会议
输出

论文套用句式
项目团队共X人，沟通渠道数=n(n-1)/2=X(X-1)/2=X条，沟通复杂度较高。我通过沟通需求分析制定沟通管理计划：对高层采用交互式沟通（月度例会）；对团队成员采用推式沟通（周报邮件）；对外部相关方采用拉式沟通（共享文档平台），确保信息及时准确传递。

项目风险管理 — 7个过程
规划5 · 执行1 · 监控1 · 风险登记册贯穿始终
解决"什么可能出错/出好，怎么提前应对"的问题。风险 = 威胁（负面）+ 机会（正面）。威胁应对：回避/转移/减轻/接受/升级；机会应对：开拓/分享/提高/接受/升级。定性分析（P×I矩阵）必做；定量分析（蒙特卡罗）可选。风险登记册是核心文件，贯穿所有风险过程。
规划 8.1 规划风险管理 制定如何开展风险管理活动的方法
【干什么】 制定风险管理计划，规定：风险管理的方法论、风险的概率和影响定义标准（如影响"高"=工期延误>2周）、风险阈值、风险分类方式（RBS）等。是风险管理的"元计划"。
记忆钩：输出只有一个=风险管理计划（定义了风险管理的规则）。
输入
项目章程项目管理计划（所有子计划）项目文件（干系人登记册）事业环境因素组织过程资产
工具与技术
专家判断数据分析（干系人分析）会议
输出
风险管理计划（方法/角色/预算/时间/风险类别/概率影响定义/报告格式）
论文套用句式
我制定了风险管理计划，明确了风险概率和影响的评级标准（影响"高"=工期延误>2周或成本超支>10%），规定了风险登记册的维护责任人（项目经理）和更新频率（每两周一次），确保风险管理工作规范有序开展。
规划 8.2 识别风险 列出所有可能影响项目的风险事件
【干什么】 识别所有可能对项目目标产生影响的风险（威胁+机会），建立风险登记册。常用工具：头脑风暴（快速发散）、SWOT分析（全面识别内外部）、核对单（参考历史项目）、提示清单（PESTLE/TECOP框架）。
记忆钩：识别风险输出两个：风险登记册+风险报告（整体风险评估）。
输入
项目管理计划（所有子计划）项目文件（假设日志/成本/进度/质量/需求/风险/干系人）采购文件协议事业环境因素组织过程资产
工具与技术
专家判断数据收集（头脑风暴/核对单/访谈）数据分析（根因/假设与制约/SWOT/文件）人际关系与团队技能（引导）提示清单（PESTLE/TECOP）会议
输出
风险登记册（描述/触发条件/风险责任人）风险报告（整体项目风险敞口）项目文件更新（假设日志）
论文套用句式
在识别风险阶段，我组织团队、专家及甲方代表通过头脑风暴和SWOT分析，共识别风险X项，包括技术风险X项、进度风险X项、外部风险X项，全部记录于风险登记册，同时形成风险报告描述整体风险状况，作为后续定性分析的输入。
规划 8.3 实施定性风险分析 用概率×影响矩阵确定风险优先级
【干什么】 对识别出的风险用概率与影响矩阵（P×I）进行主观评分，确定高/中/低优先级，决定哪些风险需要重点关注。这是必做步骤（定量分析是可选的）。
记忆钩：定性=主观评分排优先级（快速）；定量=数值模拟（精确但复杂）。
输入
项目管理计划（风险管理计划）项目文件（假设日志/风险登记册/干系人登记册）事业环境因素组织过程资产

【干什么】 对高优先级风险进行数值化分析，常用蒙特卡罗模拟（随机模拟千次以上，得出完工日期/成本的概率分布）和决策树分析（计算各决策路径的期望值）。本过程是可选的（资金充足的大型项目才做）。
记忆钩：定量=蒙特卡罗/决策树/灵敏度/影响图；输出只更新风险报告。
输入

规划 8.5 规划风险应对 为高优先级风险制定具体应对策略
【干什么】 制定具体的应对策略，减少威胁/增强机会。应对策略是最高频考点：
威胁：回避（消除风险）/转移（转给第三方）/减轻（降低概率或影响）/接受（主动留储备/被动啥不做）/升级（超出权限上报）

项目采购管理 — 3个过程
规划1 · 执行1 · 监控1 · 合同类型是考试重点
解决"哪些东西需要外部采购/外包"的问题。合同类型是高频考点：固定总价FFP（风险在卖方，买方最安全，适合需求明确）；成本补偿CR（风险在买方，适合需求不明确）；时间与材料T&M（风险共担，适合外包劳务）。自制/外购分析是规划采购的核心决策工具。

项目干系人管理 — 4个过程
启动1 · 规划1 · 执行1 · 监控1 · 参与度矩阵是核心
解决"谁会影响或被影响这个项目，怎么让他们支持"的问题。权力/利益方格：高权力高利益→重点管理；高权力低利益→令其满意；低权力高利益→随时告知；低权力低利益→监督即可。干系人参与度矩阵5级：不知晓→抵制→中立→支持→领导（C=当前/D=期望，推动C向D靠拢）。
启动 10.1 识别干系人 识别所有相关方，建立干系人登记册
【干什么】 在项目启动时（也需持续进行）识别所有可能影响项目或被项目影响的人/组织，分析他们的权力、利益、影响力，建立干系人登记册。识别越早、越全面，后续管理越主动。
记忆钩：干系人登记册是"识别干系人"的核心输出，后续所有干系人过程都要用到它。
输入
项目章程项目管理计划（沟通/干系人参与计划）项目文件（变更日志/问题日志/需求文件）协议事业环境因素组织过程资产
工具与技术

输出
干系人登记册（基本信息/评估/分类/参与度）变更请求项目管理计划更新（沟通/干系人参与计划）
论文套用句式
在项目启动阶段，通过干系人分析识别干系人共X类，包括（甲方业务部门/IT部门/最终用户/监管机构/外部供应商）等，建立干系人登记册。运用权力/利益方格将（甲方项目负责人）归入"重点管理"象限，将（最终用户）归入"随时告知"象限，为制定差异化参与策略提供依据。
规划 10.2 规划干系人参与 制定提升干系人参与度的策略
【干什么】 针对各干系人制定干系人参与计划，核心工具是干系人参与度矩阵，标注每个干系人的C（当前参与度）和D（期望参与度），差距就是需要采取行动的地方。
记忆钩：参与度5级：不知晓(U)/抵制(R)/中立(N)/支持(S)/领导(L)；C到D的距离=管理工作量。

【干什么】 主动与干系人沟通，管理其期望，解决其关切，处理问题，提升其参与度。重点处理"抵制"型干系人——找出反对根本原因，通过谈判和沟通争取支持。
